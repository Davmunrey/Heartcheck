# Internacionalización

## Idiomas soportados

- Español (`es`) — idioma principal del producto.
- Inglés (`en`) — idioma de fallback en el backend.

Cualquier endpoint del API que devuelva texto orientado al usuario (mensajes y disclaimer en `AnalysisResponse`, contenido educativo) selecciona idioma a partir de la cabecera estándar `Accept-Language`. Si no se envía, se usa inglés.

## Cliente: cómo enviar el idioma

| Cliente | Cómo se manda `Accept-Language` |
|---------|--------------------------------|
| SPA Vite | [`web/src/api/heartscan.js`](../web/src/api/heartscan.js) inyecta `navigator.language` en cada request. |
| Web estática | [`web_public/app.html`](../web_public/app.html) lo añade en el `fetch` de análisis. |
| Mobile Flutter | [`apps/mobile/lib/core/api_client.dart`](../apps/mobile/lib/core/api_client.dart) acepta `acceptLanguage` (parámetro), por defecto `en`; los pantallas usan el locale actual. |

## Backend

- Errores HTTP llevan `error_code` estable (p. ej. `INVALID_CONTENT_TYPE`, `QUOTA_EXCEEDED`); el cliente puede traducir por código sin depender del `message`.
- Mensajes orientados al usuario en `analysis_pipeline._messages_for_class` se ramifican por `Accept-Language`.

## Flutter

- Fuentes: [`apps/mobile/lib/l10n/app_en.arb`](../apps/mobile/lib/l10n/app_en.arb) y [`app_es.arb`](../apps/mobile/lib/l10n/app_es.arb).
- Generación: `flutter gen-l10n` (config en [`apps/mobile/l10n.yaml`](../apps/mobile/l10n.yaml)).
- Regla: ninguna cadena visible al usuario fuera de los archivos ARB (salvo tests).

## Contenido educativo

- [`apps/ml-api/app/knowledge/ecg_topics_en.yaml`](../apps/ml-api/app/knowledge/ecg_topics_en.yaml) y `_es.yaml` deben tener la misma estructura de `id`.
- El endpoint `/api/v1/education` selecciona archivo por la query/locale.

## Pruebas mínimas

- `pytest apps/ml-api/tests/test_analyze.py::test_analyze_success` cubre `Accept-Language: en`.
- Añadir, ante cambios de copy, un test que verifique `message` español al pasar `Accept-Language: es`.
